## On A Post-It

Audited five tools (webfetch, fastpdf, arxiv query, ArxivAPI, WikipediaAPI) ahead of publication. Two bugs caught: `tool_fastpdf.py` had its CLI body at module level instead of a `main()` function with `__name__` guard; `ArxivAPI.py` comment referenced "arvix" instead of "arxiv". Created `requirements.txt` with pip dependencies annotated per source file. Discovered the git repo had no `user.name`/`user.email` configured — locally set both and enabled `user.useConfigOnly true` to prevent silent OS username fallback on commits.

---

## Fixes

### tool_fastpdf.py — Module-level CLI body

The argument parser and execution logic (lines 33-79) ran at module scope. Safe under `if __name__ == "__main__":` but not importable as a library — any `from tool_fastpdf import ...` would trigger `argparse.parse_args()`. Wrapped in a `main()` function with the guard, enabling both `python tool_fastpdf.py` and programmatic import without side effects.

### ArxivAPI.py — Typo

Comment on line 15 read "pip install arvix" (misspelling of the package name). Corrected to "arxiv".

### requirements.txt

Six packages, alphabetical, with inline `# source_file` comments:

```
arxiv               # ArxivAPI
pydantic            # ArxivAPI, WikipediaAPI
pymupdf4llm         # fastpdf
pyyaml              # fastpdf, hool_query_arxiv
requests            # webfetch, WikipediaAPI
trafilatura         # webfetch
```

### Git config — Local-only identity

No `user.name` or `user.email` was set anywhere (local or global). Commits would fall back to OS username/hostname — a silent failure for GitHub publication. Set local `user.name` and `user.email` for this repo, and enabled `user.useConfigOnly true` globally to hard-fail on any repo without explicit config, eliminating the fallback entirely.
